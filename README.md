# Advanced Data Analytics & Big Data Storage and Processing Project

# Project Overview

This repository contains the implementation and documentation for the assessment project required for the MSc in Data Analytics at CCT College Dublin. The project includes data‑engineering tasks (Spark/Hadoop processing), SQL/NoSQL storage, and database performance evaluation of two systems using YCSB, as well as advanced analytics involving tweet and stock preprocessing, sentiment analysis, and multi‑horizon time‑series forecasting using both financial data and social‑media text.

One of the goals of this project is to analyse a combined dataset of 10,000 tweets and daily stock price data (Jan–Dec 2020) for 38 companies and to complete two main tasks:

1. Preprocess, store, and perform analytical operations on the data using distributed and database technologies.

2. Develop forecasting models capable of predicting 1‑day, 3‑day, and 7‑day CLOSE prices for at least five companies/tickers.

The project includes several tasks:

- Data processing using Spark and Hadoop HDFS

- SQL/NoSQL storage for processed data

- Database benchmarking using YCSB

- Real-time analytics using Spark

- Sentiment extraction from tweet text

- Time‑series forecasting using neural networks and autoregressive models

- A dynamic, interactive dashboard for visualising forecast results

## Big Data Storage and Processing

For the stock-price dataset, I implemented a storage and data-processing solution using Hadoop (HDFS) for storage, Hive Metastore for metadata management, and Spark/Spark SQL as the distributed computation engine. Hive was not used as a query engine; only its Metastore component was utilised, as it provides structural metadata such as table schemas, column names, data types, file locations, and partitions. 

A Hive-compatible table was then created in HDFS in Parquet format using the spark.sql() function by submitting a SQL DDL statement. The table was partitioned by ticker, and the processed dataframe was inserted into this table using the intertInto(“stock_prices”) method. I then verified data integrity by checking for null values using Spark SQL. Additional analytical queries were performed using Spark SQL.

Why Hive Metastore + Spark SQL + Parquet + Hadoop HDFS?

1. Structured nature of stock-price data: The dataset fits naturally into a tabular schema with fields such as ticker, date, open, etc.
2. Partitioning improves performance: Partitioning by ticker speeds up queries that filter by ticker
3. Straightforward configuration: Integrating Spark with Hive Metastore requires minimal configuration (copying hive-site.xml into Spark configuration directory)
4. Metadata management without Hive execution:  Hive Metastore stores table metadata and enables SQL-style interactions without requiring Hive’s execution engine
5. Parallel execution in Spark: Although Hadoop MapReduce was not used and cannot be directly compared with Spark, the latter demonstrated parallel execution by creating multiple tasks

Why MongoDB for Tweets?

MongoDB was selected because its flexible schema allows documents to have varying structures and lengths, and schemas evolve. Tweets contain variable-length text; this characteristic is ideally suited for MongoDB’s document model.  MongoDB also provides indexes to support efficient queries, enabling faster retrieval of specific tickers or date ranges without scanning the entire collection. Text data with emojis is easily stored in MongoDB thanks to its full UTF-8 support for international characters. MongoDB also supports date and int types.

## Performance Analysis: MySQL vs MongoDB using YCSB

All observations were derived from YCSB output across workloads A, B, and C. Each experiment was executed once due to time constraints, although repeating the test and averaging results could improve reliability.  The virtual machine used for the experiments was configured with 4 vCPUs and 8 GB RAM.

To ensure fairness, both MySQL and MongoDB remained running for the entire duration of the test. This reflects a real production system, where databases are rarely restarted unless necessary. After each experiment, the benchmark tables were deleted and reloaded.

All results are stored in an Excel file named YCSB comparison in this repository.

Overall conclusions:

1. MongoDB outperforms MySQL across all three workloads, especially when writes are involved
2. Workload C shows MySQL’s best performance
3. MongoDB was more suitable for the evaluated concurrent, write-intensive workloads
4. MySQL’s performance is stable for heavy-read workloads, but does not scale well with concurrent writes

## Real-time analytics

I implemented a real-time data ingestion and analytics system using PySpark, Spark Streaming, and a public weather API (OpenWeatherMap). 

Weather data is continuously collected for multiple cities and written as JSON events into a local folder. Spark Streaming monitors this folder, ingests new weather events, and performs:

1. Real-time extraction of temperature, humidity, wind speed, and weather conditions
2. Window-based temperature averages calculation
3. Threshold-based alerts (low temperature, high wind)
4. Streaming results are written to a memory sink, which stores the output in an in-memory table, allowing querying of the table with Spark SQL

## Advance Data Analytics

This project investigates whether public sentiment extracted from tweets can enhance short-term stock price forecasting. Using daily tweets and historical closing prices for five major companies across 2020, financial time-series modelling is integrated with Natural Language Processing to analyse the relationship between social sentiment and stock prices. Two forecasting approaches are compared: an autoregressive model (ARIMAX) and a neural network (LSTM). 

The tweets dataset contains 10.000 social-media posts from 2020 referencing companies/tickers. Before sentiment scoring and merging this dataset with financial time-series data, data profiling and cleaning were performed to ensure reliability and temporal consistency.

Tweet activity in the dataset is highly uneven across companies. A frequency plot (see below) shows a strong skew, with TSLA receiving far more tweets than others, followed by AAPL, BA. This imbalance is important because companies with more text data produce more stable and meaningful sentiment scores.  


<img width="940" height="516" alt="image" src="https://github.com/user-attachments/assets/956171b9-d574-4549-99c6-0bba8588385e" />


For the stock-price dataset, a data-profiling step was performed to ensure the structure and quality of the data were suitable for subsequent modelling. The checks were similar to those used for the tweet’s dataset: validating column types, detecting missing values, checking for duplicates, and converting the date column into a proper datetime index. All five tickers covered the full 2020 period. Gaps appeared only on weekends and holidays, which is expected for financial markets.

Both ARIMA and LSTM models were applied directly to the training-day data without filling weekend or holiday gaps. Dhyani et al. (2020) used ARIMA on daily stock prices without gap-filling, and Nelson et al. (2017) trained an LSTM model on finance data without adding synthetic values.

A comparison of closing prices (see plot below) shows clear differences in behaviour across companies:

- AAPL and AMZN show steady upward trends over the year
- TSLA exhibits strong growth and high volatility
- BA experiences a crash in March and remains unstable afterwards
- DIS drops early in the year but slowly recovers

These patterns indicate non-stationary behaviour, meaning differencing will be required for ARIMAX models.


<img width="940" height="469" alt="image" src="https://github.com/user-attachments/assets/f47da79c-8c1f-41e5-9fe6-1f0c3e355074" />


Two sentiment tools were used:
- VADER, which is specifically designed for social-media language and captures expressive features such as emojis, intensity markers, ALL CAPS words, etc. (Hutto and Gilbert, 2014)
- TextBlob, included as a simpler baseline for comparison

Both methods produced a sentiment score for each tweet. These scores were then aggregated by date and ticker, and later merged with the stock price dataset. Alongside sentiment, the daily tweet count was included as an additional predictive feature. I added the number of tweets because the dataset is quite small, and having an extra piece of information could help the model make better predictions.  Also, if people tweet a lot about a company on a given day, it often means more attention, which might affect the stock price.  So, tweet count could help the model understand the situation better.

Since the series are non-stationary and there is no meaningful seasonal component, a non-seasonal ARIMAX model is the most appropriate approach for forecasting. The “X” in ARIMAX means the model includes external predictors in addition the stock’s own past values.

In the ARIMAX function, a loop iterated over multiple combinations of p (autoregressive order) and q (moving average order) parameters. For each configuration the model was fitted and its evaluation metrics along with 1-day, 3-day and 7-day forecasts were stored inside dictionaries. This allowed to compare all tested configurations and select the best performing ARIMAX parameters.

LSTM was selected because it is specifically designed for sequential data and can learn non-linear patterns that traditional autoregressive models such as ARIMAX cannot capture (Yamak et al., 2019). Following the approach in Yamak et al., the model was implemented using TensorFlow/Keras which offers a simple and flexible API for constructing LSTM architectures. Hyperparameter optimisation was performed using Keras Tuner, a framework tightly integrated with Keras that automates the search process for optimal hyperparameters and identifies the best-performing configuration (O’Malley et al., 2019).

RMSE and MAE were used to evaluate ARIMAX and LSTM models. MAE shows the typical error size, while RMSE highlights if large errors occur. Train metrics show how well the model fits known data and test metrics show how well it generalises to unseen data. A large gap between training and test errors signals overfitting, while high error on both sets suggests underfitting. A small gap can indicate better generalisation, but it does not guarantee that the model captures meaningful patterns (Scikit-learn, 2024; (Montesinos López et al., 2022; McQuaid, 2025).
 
ARIMAX:

- AAPL, TSLA, and DIS show a large train-test gap, meaning ARIMAX seems to generalise poorly
- TSLA and BA exhibit very large test errors, likely due to their stronger volatility and non-linear behaviour, which ARIMAX cannot capture
- Only AMZN achieved a reasonably balanced performance, likely due to its better linear structure in its series.
  
LSTM:

- Consistently achieved lower test errors than ARIMAX, indicating a better ability to capture patterns in stock price data and sentiment effects
- LSTM produced better results using learning-rate reduction and L2 regularisation, suggesting these features helped prevent overfitting and improve generalisation

Tufte’s Principles in the Dashboard:

The charts were designed to be clear and easy to read. They avoid unnecessary decorations and only show what is needed to understand the results. Colours are used in a simple and consistent way so the reader can easily see the difference between actual and predicted values.
All tickers use the same style of graphs, which makes it easier to compare them. The graphs also use subtle grid lines and clean backgrounds, so the focus stays on the data, not on extra visual elements.  
Overall, the dashboard follows Tufte’s idea of “show the data clearly without adding clutter”.
